# ログイン処理の流れ

## 全体フロー

```
ブラウザ                    Spring Security                  DB
   │                              │                          │
   │  POST /users/login           │                          │
   │  { email, password }         │                          │
   │─────────────────────────────>│                          │
   │                              │                          │
   │               JwtAuthenticationFilter が先に動く        │
   │               (Authorizationヘッダーなし → スルー)      │
   │                              │                          │
   │                    UserController.login()               │
   │                              │                          │
   │               AuthenticationManager.authenticate()      │
   │                              │  findByEmail(email)      │
   │                              │─────────────────────────>│
   │                              │  User取得                │
   │                              │<─────────────────────────│
   │                              │  BCryptでpassword照合    │
   │                              │                          │
   │                    JwtService.generateToken()           │
   │                              │                          │
   │  { token: "eyJ...", name }   │                          │
   │<─────────────────────────────│                          │
```

## ① フロントエンド（LoginForm.jsx）

```js
const res = await fetch('/users/login', {
    method: 'POST',
    headers: {'Content-Type': 'application/json'},
    body: JSON.stringify({email: form.email, password: form.password}),
})
```

メールアドレスとパスワードをJSONでPOST送信する。

## ② JwtAuthenticationFilter（スルーされる）

`SecurityConfig` でフィルターが `UsernamePasswordAuthenticationFilter` の**前**に挟まっている。

```kotlin
.addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter::class.java)
```

このフィルターはリクエストの `Authorization` ヘッダーを見る。

```kotlin
val authHeader = request.getHeader("Authorization")

if (authHeader == null || !authHeader.startsWith("Bearer ")) {
    filterChain.doFilter(request, response)  // ← ヘッダーがないのでスルー
    return
}
```

ログイン時はトークンを持っていないのでヘッダーなし → **何もせず次の処理へ通す**。

---

## ③ SecurityConfig（permitAll なので認証不要）

```kotlin
.requestMatchers("/users/register", "/users/login").permitAll()
```

`/users/login` は認証なしで通過できるように設定済みなので、未ログインでもアクセスできる。

## ④ UserController.login()

```kotlin
@PostMapping("/login")
fun login(@RequestBody request: LoginRequest): ResponseEntity<Any> {
    return try {
        authenticationManager.authenticate(            // ← ここで認証
            UsernamePasswordAuthenticationToken(request.email, request.password)
        )
        val userDetails = userDetailsService.loadUserByUsername(request.email)
        val token = jwtService.generateToken(userDetails)   // ← JWT発行
        val user = service.findByEmail(request.email)!!
        ResponseEntity.ok(mapOf("token" to token, "name" to user.username))
    } catch (e: BadCredentialsException) {
        ResponseEntity.status(HttpStatus.UNAUTHORIZED)
            .body(mapOf("error" to "emailアドレスかパスワードが間違っています"))
    }
}
```

---

## ⑤ AuthenticationManager が認証する内部の動き

`authenticationManager.authenticate(...)` を呼ぶと、内部で以下が動く。

```
AuthenticationManager
    └── DaoAuthenticationProvider（AppConfigで登録したBean）
            │
            ├── CustomUserDetailsService.loadUserByUsername(email)
            │       └── UserRepository.findByEmail(email)
            │               → DBからUserを取得
            │               → emailが見つからない場合 → UsernameNotFoundException
            │
            └── BCryptPasswordEncoder.matches(入力パスワード, DBのハッシュ)
                    → 一致しない場合 → BadCredentialsException
```

**ポイント:** パスワードはDBに平文ではなくBCryptハッシュで保存されている。`matches()`
は入力値をハッシュ化して比較するので、元の値を復元することなく照合できる。

認証に失敗すると `BadCredentialsException` がスローされ、`catch` ブロックで401エラーを返す。

## ⑥ JWT発行（JwtService.generateToken）

認証成功後、JWTトークンを生成する。

```kotlin
private fun createToken(claims: Map<String, Any>, subject: String): String {
    val now = Date()
    val expiration = Date(now.time + 1000 * 60 * 60 * 24) // 24時間有効

    return Jwts.builder()
        .setClaims(claims)
        .setSubject(subject)      // ← emailアドレスをsubjectに格納
        .setIssuedAt(now)
        .setExpiration(expiration)
        .signWith(secretKey, SignatureAlgorithm.HS256)  // ← 秘密鍵で署名
        .compact()
}
```

JWTは `ヘッダー.ペイロード.署名` の3パート構造。

```
eyJhbGciOiJIUzI1NiJ9          ← ヘッダー（アルゴリズム情報）
.eyJzdWIiOiJ0ZXN0QC4uLiJ9    ← ペイロード（email・有効期限など）
.SflKxwRJSMeKKF2QT4fwpMeJ... ← 署名（秘密鍵で改ざん検知）
```

## ⑦ フロントエンド（ログイン後の状態管理）

```js
localStorage.setItem('token', data.token)    // ← ブラウザに保存（ページ更新しても残る）
localStorage.setItem('username', data.name)

onLoginSuccess(data.token, data.name)        // ← App.jsx の state を更新
```

`App.jsx` 側:

```js
const handleLoginSuccess = (newToken, name) => {
    setToken(newToken)
    setLoggedInUser(name)   // ← ヘッダーの表示が切り替わる
    setCurrentPage('board') // ← 掲示板画面に遷移
}
```

## ⑧ ログイン後のリクエスト（JWT認証）

ログイン後に保護されたAPIを叩く際は、`Authorization: Bearer {token}` ヘッダーを付ける。
すると今度は `JwtAuthenticationFilter` がトークンを検証して認証状態をセットする。

```kotlin
val jwt = authHeader.substring(7)               // "Bearer " を除いたトークン
val username = jwtService.extractUsername(jwt)  // ペイロードからemail取得
// ...
if (jwtService.isTokenValid(jwt, userDetails)) {
    // SecurityContextHolder に認証情報をセット → 認証済みとして扱われる
    SecurityContextHolder.getContext().authentication = authToken
}
```

---

## まとめ

```
ログイン時:   email+password → AuthenticationManager → BCrypt照合 → JWT発行 → localStorage保存
以降のAPI:   JWT付きリクエスト → JwtAuthenticationFilter → 署名検証 → 認証済みとして通過
```
